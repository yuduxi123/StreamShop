一、一句话概括项目
================================================================================

StreamShop 是一个"电商直播"App，类似抖音的电商版——
用户刷短视频时看到商品，可以边看边买；主播可以开直播带货。

================================================================================
二、技术栈（老师必问："你用了什么技术？"）
================================================================================

  前端（Android）: Java, Android SDK
    - ExoPlayer（视频播放）
    - CameraX/Camera1（摄像头采集）
    - RTMP 推流（pedroSG94 库）
    - MPAndroidChart（图表）
    - Glide（图片加载）
    - OkHttp + JWT Token 认证
    - WebSocket（即时通讯、弹幕）
    - ViewPager2 + RecyclerView（滑动切换视频）

  后端（服务器）: Node.js + Express + TypeScript
    - JSON 文件存储（不用数据库）
    - JWT 身份认证
    - WebSocket 实时通信
    - node-media-server（RTMP 直播服务器）

  管理后台: React + Vite + Ant Design

三、项目架构（老师必问："数据是怎么流转的？"）
================================================================================

  用户操作 → Android Activity/Fragment
    → ViewModel（业务逻辑层）
      → ApiService（HTTP 请求，OkHttp）
        → 后端 Express 路由
          → StorageService（读写 JSON 文件）
            → JSON 文件（数据库）

  举例：用户刷视频
    1. FeedFragment 调用 viewModel.loadVideos()
    2. FeedViewModel 调用 ApiService.getVideos()
    3. OkHttp 发送 GET http://10.0.2.2:3000/api/videos?page=1&limit=10
    4. 后端 video.ts 路由收到请求
    5. StorageService 从 videos.json 读取视频数据
    6. 返回 JSON 给 Android
    7. ViewModel 更新 LiveData → Fragment 刷新界面
    8. ExoPlayer 开始播放视频

  关键模式: MVVM（Model-View-ViewModel）
    - Model: 数据（Video, Product, Order 等 POJO 类）
    - View: 界面（Activity/Fragment + XML 布局）
    - ViewModel: 中间层（管理数据加载、状态）

================================================================================
四、核心功能模块（按重要性排序）
================================================================================

【模块 1】短视频 Feed 流 ⭐⭐⭐⭐⭐
  - 类似抖音的上下滑动切换视频
  - ViewPager2 + RecyclerView.Adapter 实现（不是 FragmentStateAdapter，
    因为 Fragment 方案有生命周期崩溃问题）
  - 每个"页"是一个 ViewHolder，包含 ExoPlayer 播放器
  - 只有当前可见的视频在播放，其他暂停
  - 视频上有：作者头像、商品卡片、点赞/评论/分享按钮

  数据流:
    FeedFragment → FeedPagerAdapter → VideoViewHolder
      → ExoPlayer.setMediaItem(url) → 播放

  预加载: 只加载前一个和后一个（offscreenPageLimit = 1）

【模块 2】直播推流 ⭐⭐⭐⭐⭐
  - 主播开播: 摄像头采集 → 编码 → RTMP 推流到服务器
  - 观众看播: 从服务器拉流播放

  推流流程:
    摄像头(Camera1) → RtmpCamera1(pedroSG94库)
      → H.264/AAC 编码
        → RTMP 协议推送到 rtmp://10.0.2.2:1935/live/房间ID

  关键技术:
    - Camera1 API（非 Camera2，因为模拟器兼容性）
    - prepareVideo(1280, 720, 30fps, 1200kbps, 270°旋转)
      - 1280×720 = 分辨率
      - 30 = 帧率
      - 1200*1024 = 码率 1.2Mbps
      - 270 = 旋转角度（摄像头横向→竖屏）
    - RTMP 服务器使用 node-media-server

【模块 3】电商功能 ⭐⭐⭐⭐
  - 商品绑定视频（video_products.json 记录关联关系）
  - 购物车 CRUD
  - 下单 → 支付 → 订单追踪
  - 视频播放时右下角弹出商品卡片，点击可查看详情/加购

  数据结构:
    videos.json  ←→  video_products.json  ←→  products.json
                                        ←→  carts.json
                                        ←→  orders.json

【模块 4】运营分析 Dashboard ⭐⭐⭐
  - KPI 卡片: 总播放/总点赞/总评论（仅当前用户）
  - 视频表现: 横向柱状图，按播放量降序（MPAndroidChart 的 HorizontalBarChart）
  - 转化漏斗: 商品曝光 → 挂载视频数 → 关联商品数 → 商品总数
  - GMV 排行: 全平台按 GMV 排名（非当前用户）
  - 热门商品: 全平台按被关联视频次数排名

  重要: 所有数据从真实 JSON 文件实时计算，没有假数据（除早期的 seed 数据）

【模块 5】即时通讯与弹幕 ⭐⭐⭐
  - WebSocket 实现实时消息
  - 弹幕: 发送后存储到后端，WebSocket 广播给所有观众
  - 私信: 用户之间的文字聊天

【模块 6】用户系统 ⭐⭐⭐
  - JWT Token 认证（登录后获得 Token，后续所有请求带 Token）
  - AuthInterceptor: OkHttp 拦截器，自动在每个请求头加上 Bearer Token
  - 注册: 手机号 ≥11 位，密码必须含字母+数字

五、老师高频问题 & 回答模板
================================================================================

Q1: 为什么选择 JSON 文件存储而不是数据库？
A: 这是一个学习项目。JSON 文件方案的好处：
   - 零配置，不需要安装 MySQL/MongoDB
   - 可以直接打开文件查看数据，方便调试
   - Git 可以追踪数据变化
   - StorageService<T> 泛型类封装了所有 CRUD 操作
   - 缺点：不适合高并发、不支持复杂查询
   如果是生产环境，会用 MySQL + Redis

Q2: 为什么用 ViewPager2 + RecyclerView.Adapter 而不是 FragmentStateAdapter？
A: FragmentStateAdapter 在销毁重建时会崩溃（Design assumption violated）。
   根本原因是 Fragment 生命周期与 ViewPager2 的耦合。
   改为 RecyclerView.Adapter + ViewHolder 后：
   - ViewHolder 生命周期由 RecyclerView 控制，更可控
   - 不需要 childFragmentManager，消除 Fragment 相关崩溃
   - 每个 ViewHolder 独立管理自己的 ExoPlayer

Q3: 推流时为什么选择 Camera1 而不是 Camera2？
A: Camera2 + OpenGL 在模拟器上会直接闪退。Camera1 API 更成熟稳定。
   虽然在真机上 Camera2 效果更好（支持更多控制），但开发阶段优先稳定性。

Q4: 直播的 RTMP 推流是如何工作的？
A: 1. Camera1 采集摄像头画面
   2. pedroSG94 库的 RtmpCamera1 将画面编码为 H.264，音频编码为 AAC
   3. 通过 RTMP 协议推送到 node-media-server（端口 1935）
   4. 服务器将流转发给所有观看者
   5. 观看者通过 HLS/FLV 协议拉流观看

Q5: 如何处理多个视频同时播放（音频重叠）的问题？
A: ViewPager2 的 OnPageChangeCallback 监听页面切换：
   - 只有当前选中的页面才调用 player.play()
   - 其他页面全部 player.pause()
   - 只在 onPageSelected 时处理，确保只有一个播放器活跃

Q6: 弹幕是怎么实现的？
A: Android 端发送弹幕 → POST /api/danmaku（存储）
   → WebSocket 广播给同一直播间的所有观众
   → 每个观众收到后调用 DanmakuView.addDanmaku() 渲染
   去重: 发送者本地直接渲染，收到 WebSocket 消息时跳过自己的 userId

Q7: 运营分析的数据是真实的吗？
A: 全部从真实数据计算。KPI 按当前用户过滤，热门商品和 GMV 排行是全平台。
   没有使用任何 Math.random() 假数据。
   关键是后端 stats.ts 路由加了 authMiddleware，按 req.user.id 过滤。

Q8: 遇到过什么困难？怎么解决的？
A: 最大的三个困难:
   1. Fragment 生命周期崩溃 — 改用了 RecyclerView.Adapter 替代 FragmentStateAdapter
   2. 摄像头权限重复弹窗 — 用 startAttempted 标志位防止重入
   3. Camera2+OpenGL 闪退 — 回退到 Camera1，确保稳定性
   详见《项目问题与解决方案汇总.txt》

七、关键 API 接口速记
================================================================================

  视频:     GET    /api/videos?page=1      → 视频列表
  视频:     POST   /api/videos             → 创建视频（需登录）
  视频:     POST   /api/videos/:id/view    → 增加播放量
  直播:     GET    /api/live-rooms         → 直播间列表
  商品:     GET    /api/products           → 商品列表
  购物车:   POST   /api/cart               → 加入购物车（需登录）
  订单:     POST   /api/orders             → 下单（需登录）
  统计:     GET    /api/stats/dashboard    → 运营分析（需登录）
  登录:     POST   /api/auth/login         → 返回 JWT Token
  弹幕:     POST   /api/danmaku            → 发送弹幕
