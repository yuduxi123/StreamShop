一、项目概述
================================================================================

StreamShop 是一个仿抖音风格的短视频直播电商平台，包含三个子系统：

  1. Android App（Java）     — 用户端，刷视频、看直播、下单购物
  2. Express 后端（TypeScript）— REST API + WebSocket 服务
  3. React 管理后台（Vite）   — 管理员管理商品/订单/直播间/视频

核心功能：
  - 短视频上下滑动浏览（像抖音一样）
  - 视频中嵌入商品卡片，点击进入商品详情
  - 直播间实时互动（弹幕、点赞、商品讲解、优惠券推送）
  - 购物车 → 下单 → 支付完整电商链路
  - 用户系统（注册/登录/关注/头像裁剪）
  - 视频上传编辑、商品管理

3.2 认证流程（JWT 机制）
  1. 用户注册：POST /api/auth/register
     - 接收 username + password
     - password 用 bcryptjs.hash() 加密（不存明文）
     - 生成 JWT token（有效载荷：userId + role）
     - 返回 token → 客户端保存到 ApiClient.authToken

  2. 用户登录：POST /api/auth/login
     - 查找用户 → bcryptjs.compare() 验证密码
     - 生成新 token → 返回

  3. 后续请求：
     - ApiClient 的 AuthInterceptor 自动在请求头注入：
       Authorization: Bearer <token>
     - 后端 auth 中间件验证 token → 解析出 userId → 注入到 req.user

3.3 数据存储（JSON 文件系统）
  所有数据存储在 backend/src/data/*.json：
    users.json, videos.json, products.json, video_products.json,
    carts.json, orders.json, comments.json, likes.json,
    collections.json, follows.json, live_rooms.json, coupons.json,
    flash_sales.json, danmaku.json, claimed_coupons.json

  StorageService<T> 通用类提供：
    - findAll()                    → 读取全部
    - findById(id)                 → 按 id 查找
    - query(fn)                    → 按条件过滤
    - create(item)                 → 新增（自动生成 id + createdAt）
    - update(id, partial)          → 更新
    - delete(id)                   → 删除
    - paginate(page, limit, filter)→ 分页

  这种方式的优点：零依赖安装（不需要 MySQL/MongoDB），
  缺点：并发写入不安全（但教学项目足够用）。

3.4 数据丰富（Enrichment）
  后端返回数据时，会把关联数据也填充进去，减少客户端请求次数。

  例如 GET /api/videos 返回的每个视频里包含：
    {
      "id": "v001",
      "title": "...",
      "videoUrl": "...",
      "author": { "id": "u001", "username": "...", "avatarUrl": "..." },
      "products": [
        { "id": "p001", "title": "...", "price": 99, "coverUrl": "..." }
      ]
    }

  实现方式：路由处理器先查 video 表，再根据 authorId 和 video_products 表去填充。

4.2 底部导航实现（MainActivity.java）

  原理：不用系统的 BottomNavigationView，而是自定义 LinearLayout + 4 个 TextView。

  布局（activity_main.xml）：
    ├── FragmentContainerView（上方内容区，约束到导航栏顶部）
    └── LinearLayout（48dp 高，黑色背景，4 个等宽 TextView）

  Java 逻辑：
    1. 初始化 tabs[] 数组：nav_feed, nav_live, nav_cart, nav_profile
    2. 初始化 tabDestIds[][] 映射到导航图中的 destination ID
    3. 每个 tab 设置 onClickListener → selectTab(index)
    4. selectTab(index)：
       - navController.navigate(tabDestIds[index][0])  → 切换 Fragment
       - 遍历 tabs 更新样式（选中：白色/15sp/粗体；未选中：#AAAAAA/13sp/正常）
    5. navController.addOnDestinationChangedListener 监听：
       - 当通过其他方式切换页面时，同步更新底部 tab 样式

  为什么不用 BottomNavigationView？
    - Material3 的 BottomNavigationView 必须有图标才能显示文字
    - 尝试了 state_selected、代码设 ColorStateList、透明度等均无效
    - 抖音风格就是纯文字无图标，自定义方案更灵活可控

4.3 视频播放核心（VideoViewHolder / VideoPageFragment）

  这是整个 App 最核心的组件，负责单个视频的完整播放体验。

  使用技术：
    - Media3 ExoPlayer（Google 官方视频播放器）
    - PlayerView（包含播放器控件）

  视频播放流程：
    1. 创建 ExoPlayer 实例
    2. 设置 MediaItem（视频 URL）
    3. player.prepare() → player.play()
    4. 页面不可见时 player.pause()
    5. Activity 销毁时 player.release()

  ViewHolder 生命周期管理：
    - onPageSelected(position) → 播放当前页，暂停其他页
    - onPause() → 停止所有播放器
    - onDestroy() → 释放所有播放器资源

  视频界面包含的 UI 元素（fragment_video_page.xml）：
    ├── PlayerView          → 视频播放器
    ├── ImageView（封面）   → 加载时显示封面，播放后隐藏
    ├── SeekBar + 时间显示  → 进度条可拖动
    ├── 播放/暂停指示器     → 点击切换
    ├── 右侧操作按钮列：
    │   ├── 头像（圆形）    → 点击进入作者主页
    │   ├── 点赞按钮        → 红色高亮切换
    │   ├── 评论按钮        → 打开评论 BottomSheet
    │   ├── 收藏按钮        → 黄色高亮切换
    │   └── 关注按钮        → 关注/已关注切换
    ├── 底部信息：
    │   ├── 用户名
    │   ├── 视频标题
    │   └── 商品卡片列表    → 横向排列，点击打开商品详情
    └── 弹幕层（VideoDanmakuView）→ 自定义 View

  播放控制逻辑：
    - 点击视频区域切换播放/暂停
    - 播放时显示 2 秒播放图标动画后隐藏
    - 暂停时显示半透明暂停图标

4.4 商品卡片展示与视频时间点绑定（VideoViewHolder 中的产品卡片）

  每个视频可以绑定多个商品，在视频中显示为横向卡片：
    - 卡片包含：商品缩略图、标题、现价、原价（划线）、折扣率
    - 卡片底部有橙色"介绍"按钮
    - 点击卡片 → 弹出 ProductDetailBottomSheetFragment
    - 弹窗中有三个按钮：立即购买 / 加入购物车 / 收藏

  商品-视频时间点绑定（核心新功能）：
    - 商家发布视频时，可为每个商品设置绑定的时间点（mm:ss 格式输入）
    - 点击商品卡片"介绍"按钮 → 视频 seekTo 到绑定时间点 → Toast 显示商品名
    - 进度条上显示白色圆点（DotSeekBar 自定义 View）标记时间位置
    - 拖拽进度条到打点附近（3% 时长阈值）自动吸附到最近商品时间点
    - 历史商品无时间点则按视频时长均匀分布：
      - 1 个商品 → 视频中心点（50%）
      - 2 个商品 → 三等分点的第 1、第 2 个（33%、67%）
      - N 个商品 → duration × (i+1) / (N+1)

  技术实现：
    - DotSeekBar.java：继承 AppCompatSeekBar，onDraw() 用 Canvas 画白色圆点
    - VideoViewHolder 新增方法：
      - getEffectiveTimestamp() → 有 timestampMs 用原值，否则计算默认位置
      - updateProgressDots() → 计算所有商品的时间比例，设置到 DotSeekBar
      - findNearProduct() → 3% 时长阈值内找最近商品，实现拖拽吸附
    - Toast 使用自定义 View（纯文本无图标），白字深灰背景
    - 后端 VideoProduct 接口添加 timestampMs 字段，feed.logic.ts + video.ts 注入

4.5 ViewPager2 垂直滑动（FeedFragment）

  仿抖音的上下滑动切换视频：

    ViewPager2 配置：
      - android:orientation="vertical"    ← 关键：垂直滑动
      - adapter: FeedPagerAdapter
      - offscreenPageLimit: 1             ← 预加载 1 页

    FeedPagerAdapter：
      - 继承 RecyclerView.Adapter
      - inflate fragment_video_page.xml 作为每个 item
      - 创建 VideoViewHolder 管理播放器

4.6 弹幕实现（VideoDanmakuView / DanmakuView）

  自定义 View，从右到左滚动显示文字弹幕。

  原理：
    - 维护一个 List<DanmakuItem>，每个 item 有：
      text（文字）、x/y（位置）、color（颜色）、
      speed（速度）、width（文本宽度）
    - 通过 Handler.postDelayed() 每 16ms（≈60fps）触发重绘
    - 每帧：所有 item 的 x 坐标减 speed → invalidate() 触发 onDraw()
    - onDraw() 中逐个绘制带描边的文字（黑色描边 + 白色文字，保证可见）

  弹幕来源：
    1. 历史弹幕：从 API 加载（GET /api/danmaku?videoId=xxx）
       按 timestampMs 在对应播放位置释放
    2. 实时弹幕：通过 WebSocket 接收（直播间和视频均支持）

  限制：
    - 同屏最多 20 条弹幕
    - 随机 Y 坐标行（4 行）
    - 视频弹幕支持暂停/继续/跳到指定位置

================================================================================
五、WebSocket 实时通信
================================================================================

5.1 服务端（backend/src/websocket/wsServer.ts）
  结构：
    - rooms: Map<roomId, Map<WebSocket, memberInfo>>
    - 每个房间维护一个成员列表

  核心方法：
    - joinRoom(ws, member)        → 加入房间
    - leaveRoom(ws)               → 离开房间
    - broadcast(roomId, message)  → 向房间广播消息
    - getOnlineCount(roomId)      → 获取在线人数
    - pushProductChanged(...)     → 推送商品变更
    - pushCoupon(roomId, coupon)  → 推送优惠券
    - pushDanmaku(videoId, dm)    → 推送弹幕
    - pushLiveStarted(room)       → 全局广播开播

  热度值机制：
    - room.hotValue 初始值
    - 每 30 秒衰减 15%（每 2 秒减 1.5%）
    - 每次点赞/评论/弹幕增加热度

5.2 客户端（LiveWebSocketClient.java）
  连接：ws://10.17.24.7:3000/ws

  发送消息类型：
    JOIN_ROOM      → { type, roomId, userId, username }
    LEAVE_ROOM     → { type, roomId }
    SEND_COMMENT   → { type, roomId, userId, username, content }
    SEND_DANMAKU   → { type, videoId, userId, username, content, color }
    LIKE           → { type, roomId, userId }
    CLAIM_COUPON   → { type, userId, couponId }
    PING           → 保活心跳

  重连机制：
    - 断线自动重连，指数退避延迟
    - 最多重试 10 次，最大延迟 30 秒

5.3 弹幕 WebSocket（视频中）
  视频播放时，通过 WebSocket 连接房间 "video_" + videoId
  实时弹幕流程：
    用户发送 → WebSocket → 服务端存储 + 广播 → VideoDanmakuView 实时渲染

================================================================================
六、数据流转全链路
================================================================================

以"用户刷视频并购买商品"为例：

1. 用户打开 App → FeedFragment 加载
2. FeedViewModel.loadVideos() → VideoRepository.loadVideos()
3. VideoRepository 调用 ApiService.getVideos(page, limit)
4. ApiService 用 OkHttp 发 GET /api/videos?page=1&limit=10
5. 后端 video.ts 路由 → StorageService 读 videos.json
6. 后端填充 author + products（数据丰富）
7. 返回 JSON → Gson 解析为 List<Video>
8. FeedPagerAdapter 创建 VideoViewHolder → ExoPlayer 播放视频
9. 用户看到商品卡片 → 点击 → ProductDetailBottomSheetFragment
10. 用户点"加入购物车" → ApiService.addToCart()
11. POST /api/cart → 后端写 carts.json → 返回成功
12. 用户去购物车 → CartFragment → 选择商品 → 结算
13. OrderConfirmActivity → createOrder()
14. POST /api/orders → 后端创建订单 + 扣减库存
15. PaymentActivity → payOrder()
16. POST /api/orders/:id/pay → 80% 概率支付成功

八、关键代码模式讲解
================================================================================

8.1 OkHttp 同步请求模式
  本项目的 ApiService 所有方法都在子线程中同步调用：

    new Thread(() -> {
        try {
            Response response = ApiClient.getInstance().newCall(request).execute();
            // 解析 JSON
            runOnUiThread(() -> {
                // 更新 UI
            });
        } catch (IOException e) {
            // 处理错误
        }
    }).start();

  为什么不用 Retrofit？
    - 教学目的，理解底层 HTTP 请求原理
    - Retrofit 本质也是封装 OkHttp

8.2 AuthInterceptor 认证拦截器
  ApiClient 内部有一个 OkHttp Interceptor：
    - 自动在所有请求头中添加 "Authorization: Bearer <token>"
    - 如果 token 为 null 则跳过
    - 比在每个方法中手动设置更安全

8.3 重试拦截器
  当后端返回 5xx 错误时，自动重试最多 2 次，指数退避延迟：
    第 1 次重试等 1 秒，第 2 次等 2 秒

8.4 Gson 序列化反序列化
  Gson 自动将 JSON 和 Java 对象互相转换：
    JSON → Java：gson.fromJson(jsonStr, Video.class)
    Java → JSON：gson.toJson(video)

8.5 ViewBinding
  使用 Android 的 ViewBinding 代替 findViewById：
    binding = ActivityMainBinding.inflate(getLayoutInflater())
    setContentView(binding.getRoot())
    binding.navFeed.setOnClickListener(...)

8.6 ViewModel + LiveData
  用于数据与 UI 解耦：
    - ViewModel 在 Activity 重建时不销毁（屏幕旋转等）
    - LiveData 在数据变化时自动通知 UI 更新
    - 使用 postValue() 在子线程中安全更新

8.7 ExoPlayer 生命周期管理
  必须在正确的时机调用：
    - onCreate → 初始化 ExoPlayer
    - onResume/onPageSelected → player.play()
    - onPause → player.pause()
    - onDestroy → player.release()
  不释放会导致内存泄漏和后台播放。

8.8 ViewPager2 + FragmentStateAdapter
  FeedFragment 使用 ViewPager2 垂直滑动：
    - FragmentStateAdapter 为每个页面创建 Fragment
    - offscreenPageLimit=1 控制预加载数量
    - registerOnPageChangeCallback 监听页面切换触发播放

8.9 底部导航重建的原因
  为什么放弃系统的 BottomNavigationView？
    - 问题：Material3 的 BottomNavigationView 无图标时文字不显示
    - 尝试：state_selected ColorStateList、代码设 textColor、
            NavigationUI.setupWithNavController 均无效
    - 根因：Material 设计规范要求必须有图标
    - 解决：自定义 LinearLayout + TextView + NavController.navigate()
    - 好处：完全控制样式，选中/未选中的文字大小、颜色、粗细可任意定义

十、AIGC 内容生成详解
================================================================================

10.1 概述
  项目集成了 DeepSeek 大语言模型，为商家提供 AI 辅助内容创作能力。
  支持 5 种内容生成类型，覆盖视频发布和直播带货场景。

10.2 技术选型：为什么用 DeepSeek？
  - API 完全兼容 OpenAI SDK（openai npm 包），零迁移成本
  - 国内可直接访问，无需代理
  - 中文理解和生成能力出色（中文电商场景的标题/卖点/话术）
  - 价格远低于 GPT-4（约 1/10）
  - 替换方便：改 AI_BASE_URL 和 AI_MODEL 环境变量即可换其他兼容模型

10.3 后端实现（ai.service.ts）

  DeepSeek API 调用：
    - 使用 openai npm 包（OpenAI 兼容 SDK）
    - baseURL: https://api.deepseek.com/v1
    - 模型: deepseek-chat（可配置）
    - temperature: 0.8（平衡创造性和一致性）
    - max_tokens: 600（足够生成 200 字以内中文）

  5 种 Prompt 模板（全部为中文，绑定电商场景）：

    1. video_title（视频标题生成）
       输入: productName, productDescription, productPrice, productCategory, videoContent
       输出: 5 个候选标题（15 字以内），换行分隔
       用途: 视频编辑页"AI生成标题"按钮 → AlertDialog 列表选择

    2. product_selling_points（商品卖点提炼）
       输入: productName, productDescription, productPrice, productCategory
       输出: 5 条卖点（30 字以内），换行分隔
       用途: 视频编辑页"AI生成卖点"按钮 → 弹窗展示 + 一键复制

    3. product_description（商品描述生成）
       输入: productName, productDescription, productPrice, productCategory
       输出: 80-150 字详细描述
       用途: 商家发布商品时辅助写描述

    4. live_commentary（直播讲解话术）
       输入: productName, productDescription, productPrice, productCategory, originalPrice
       输出: 30 秒口播脚本（80-120 字），口语化风格
       用途: 主播直播时参考讲解脚本

    5. product_recommendation（商品推荐语）
       输入: productName, productDescription, productPrice, productCategory
       输出: 60-100 字推荐文案
       用途: 商品详情弹窗"AI生成推荐语"按钮 → 弹窗展示 + 一键复制

10.4 后端路由（ai.ts）

  GET /api/ai/status
    - 返回 { configured: true/false }
    - configured = AI_API_KEY 已设置且非空且非占位符
    - Android 端据此决定是否显示 AI 按钮

  POST /api/ai/generate（需认证）
    - 请求体: { type: string, context: object }
    - type 校验: 必须为 5 种允许值之一，否则返回 400
    - context: 包含商品信息的键值对（productName, productPrice 等）
    - 调用 ai.service.ts 的 generateText()
    - 返回: { text: string, type: string }
    - 错误处理: 401（API Key 无效）、429（频率限制）、500（通用错误）

10.5 Android 端集成

  ApiService（API 层）:
    - isAiConfigured(Callback<Boolean>) → GET /api/ai/status
      控制 AI 按钮的可见性
    - generateAiContent(String type, Map<String, Object> context, Callback<String>)
      → POST /api/ai/generate → 返回生成文本

  VideoEditActivity（视频编辑页）:
    - "AI生成标题"按钮（video_edit_ai_title）
      → 收集表单中的商品名/价格/描述/视频内容
      → generateAiContent("video_title", context)
      → 解析换行分隔的 5 个标题 → AlertDialog 列表
      → 用户点击选中 → 填入标题 EditText

    - "AI生成卖点"按钮（video_edit_ai_selling_points）
      → 收集商品名/价格
      → generateAiContent("product_selling_points", context)
      → 弹窗展示卖点文本 + "复制"按钮 → ClipboardManager

    按钮可见性: onCreate 中异步检查 isAiConfigured()
    → 已配置 → VISIBLE / 未配置 → GONE

  ProductDetailBottomSheetFragment（商品详情弹窗）:
    - "AI生成推荐语"按钮（product_ai_recommend_btn）
      → 收集商品名/描述/价格/分类
      → generateAiContent("product_recommendation", context)
      → 弹窗展示推荐文案 + "复制"按钮

十二、答辩常见问题准备
================================================================================

Q1: 为什么选择这个项目？
  答：短视频+直播电商是当前最热门的商业模式（抖音/快手/TikTok）。
      本项目实现了这种模式的完整前端+后端+管理后台，技术栈全面，
      涵盖了 Android、Node.js、WebSocket、REST API 等多种技术。

Q2: 为什么后端用 JSON 文件而不用数据库？
  答：本项目是教学项目，JSON 文件存储有三大优势：
      1. 零配置——不需要安装 MySQL/MongoDB
      2. 可读性强——直接打开 JSON 文件查看数据
      3. 调试方便——出问题时直接编辑 JSON 修复
      生产环境会替换为 PostgreSQL 或 MySQL。

Q3: 安全性怎么考虑的？
  答：1. 密码用 bcrypt 哈希存储（不是明文）
      2. JWT token 认证所有写操作
      3. Android 端不存储明文密码
      4. 每个 API 操作验证用户权限（只能修改自己的资源）

Q4: 视频播放怎么优化的？
  答：1. ViewPager2 offscreenPageLimit=1，只预加载 1 个视频
      2. 不可见时暂停播放（onPageSelected 触发）
      3. 退出时释放 ExoPlayer（防止内存泄漏）
      4. Glide 加载封面后再播放视频

Q5: WebSocket 重连怎么处理的？
  答：LiveWebSocketClient 在 onFailure/onClosed 时自动重连，
      使用指数退避算法，最多 10 次，最大间隔 30 秒。

Q6: 架构用了什么设计模式？
  答：1. MVC 变体（View=XML + Activity/Fragment，数据=ViewModel + Repository）
      2. 单例模式（ApiClient）
      3. 观察者模式（LiveData + WebSocket 回调）
      4. Repository 模式（VideoRepository 封装数据源）

Q7: 项目最大的技术难点？
  答：底部导航栏改造——Material3 的 BottomNavigationView 必须有图标
      才能显示文字，而抖音风格是纯文字导航。经过多次尝试（ColorStateList、
      透明度、代码设置）均无效后，最终用自定义 LinearLayout + 4 个
      TextView 自己实现，配合 NavController 完成 Fragment 切换。

Q8: 弹幕是怎么实现的？
  答：自定义 View 继承 View，用 Canvas 绘制文字。每 16ms 通过 Handler
      触发重绘（60fps），每条弹幕的 x 坐标不断减小，模拟从右到左的
      滚动效果。文字带黑色描边保证在任何背景下都清晰可见。

Q9: 即时通讯怎么实现？
  答：直播间使用 WebSocket 实现全双工实时通信。
      客户端连接 ws://服务器地址:3000/ws，发送加入房间消息，
      服务端维护每个房间的成员列表，收到消息后向房间所有人广播。
      支持的消息类型：弹幕、评论、点赞、商品切换、优惠券推送。

Q10: 管理后台用的什么技术？
   答：React 19 + TypeScript + Vite + React Router，
       调用和 Android 相同的 REST API。

Q11: 商品-视频时间点绑定怎么实现的？
  答：这是一个完整的全链路功能——后端存储、Android 展示和交互。

  数据层：VideoProduct 表加 timestampMs 字段（毫秒），
  bind-video 端点接收并存储，feed 和 video 的 GET 端点注入到 product 对象。

  UI 层：
    - 商家发布视频时，每个商品可填 mm:ss 格式的时间点
    - 用户端商品卡片上有橙色"介绍"按钮，点击 → 视频跳转到绑定时间点
    - 进度条上白色圆点（DotSeekBar 自定义 View，Canvas 画圆）标记位置
    - 拖拽进度条到打点 3% 时长范围内自动吸附 + Toast 显示商品名

  兼容性：历史商品 timestampMs=0 → 按 videoDuration × (i+1)/(N+1) 均匀分布

  关键技术点：
    - DotSeekBar 继承 AppCompatSeekBar，onDraw() 画圆
    - findNearProduct() 遍历所有商品找最近打点（threshold = max(3%×duration, 500ms)）
    - Toast 用自定义纯文本 View（避免国产 ROM 显示 App 图标）

Q12: 商品卡片吸附滚动和 Touch 冲突怎么解决的？
  答：两个问题有因果关系——

  吸附：PagerSnapHelper 默认居中对齐，导致卡片露出半边。
  解决：自定义 StartSnapHelper，左边缘对齐，每页恰好 2 张完整卡片。

  Touch 冲突：商品卡片左右滑会触发外层 ViewPager2 切 Tab。
  最终方案：ACTION_DOWN 立刻 requestDisallowInterceptTouchEvent(true)，
  区域内所有水平滑动都归商品卡片控制；ACTION_UP/CANCEL 恢复。
  同时把 LayoutManager + SnapHelper + OnTouchListener 初始化移到构造函数，
  避免每次 bind() 重复创建导致 "OnFlingListener already set" 崩溃。

Q13: AIGC 内容生成是怎么实现的？
  答：后端接入 DeepSeek API（deepseek-chat 模型），通过 OpenAI 兼容 SDK 调用。

  流程：Android 按钮点击 → ApiService.generateAiContent(type, context)
  → POST /api/ai/generate → ai.ts 路由校验 type → ai.service.ts 构造 Prompt
  → DeepSeek API → 返回生成文本 → Android 弹窗展示。

  5 种 Prompt 模板：视频标题（5 个候选 15 字以内）、商品卖点（5 条 30 字以内）、
  商品描述（80-150 字）、直播话术（30 秒口播脚本）、商品推荐语（60-100 字）。

  选择 DeepSeek 的原因：国内直连无需代理、API 兼容 OpenAI SDK、
  中文效果好、价格低。改环境变量即可换其他模型。

Q14: UI 设计系统是怎么建立的？
  答：分 4 个 Phase 渐进式推进——

  Phase 1（设计基础）：扩展 colors.xml 从 19 色到 33 色，
  新增 styles.xml 10 种复用组件样式，将 colorPrimary 从 #FF3B30 改为
  更深邃的品牌红 #E63946。

  Phase 2（组件标准化）：顶部栏统一 56dp 黑底白字（15 个页面），
  输入框统一背景+圆角+内边距（6 个页面），
  头像收窄为 40/56/80dp 三档（28 个布局文件）。

  Phase 3（核心页面美化）：Feed 页、直播页、个人主页、直播列表、
  视频编辑页、订单页、登录注册页、聊天页共 8 大类页面逐一优化。

  Phase 4（细节打磨）：统一 elevation 层级、分割线规范、系统栏适配。

  最后批量迁移 200+ 处硬编码颜色到 @color/ 引用，
  保留 #FF3B30 在 38 处语义位置（直播红/价格红不被品牌红覆盖）。

Q15: 批量修改 200+ 处颜色值是怎么保证不出错的？
  答：关键是顺序——先扩展 colors.xml 定义所有需要的颜色（编译无影响），
  再用 replace_all 按 Pattern 分批替换（每种颜色一次操作），
  最后编译验证。

  7 种 Pattern 的替换：
  #F5F5F5→@color/background_light, #1A1A1A→@color/background_dark,
  #333333→@color/text_primary_light, #999999→@color/text_secondary,
  #666666→@color/text_secondary, #FFFFFF(文字)→@color/text_primary_dark,
  #EEEEEE→@color/divider。

  风险控制：两个颜色值语义不同即使是同一色值也不统一替换
  （如 #FF3B30 是直播红，与品牌红 #E63946 分开）；编译不通过时
  回退到文件级别逐一检查。
